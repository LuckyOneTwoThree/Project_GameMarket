# 部署与运维

## 部署方案

### 前端部署

1. **构建生产版本**

```bash
cd frontend
npm run build
```

2. **部署到 CDN**

- 使用 AWS CloudFront 或阿里云 CDN 分发静态文件
- 配置缓存策略，提高访问速度
- 配置 HTTPS，确保数据传输安全

3. **容器化部署**

```dockerfile
# Dockerfile
FROM nginx:alpine

COPY dist /usr/share/nginx/html
COPY nginx.conf /etc/nginx/conf.d/default.conf

EXPOSE 80

CMD ["nginx", "-g", "daemon off;"]
```

```nginx
# nginx.conf
server {
    listen 80;
    server_name example.com;

    location / {
        root /usr/share/nginx/html;
        index index.html;
        try_files $uri $uri/ /index.html;
    }

    location /api {
        proxy_pass http://backend:8000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
```

### 后端部署

1. **构建生产版本**

```bash
cd backend
npm run build
```

2. **容器化部署**

```dockerfile
# Dockerfile
FROM node:18-alpine

WORKDIR /app

COPY package*.json ./
RUN npm install --production

COPY dist/ ./dist/

EXPOSE 8000

CMD ["node", "dist/app.js"]
```

3. **Kubernetes 部署**

```yaml
# deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: gamemarket-backend
  namespace: gamemarket
spec:
  replicas: 3

    matchLabels:
      app: gamemarket-backend
  template:
    metadata:
      labels:
        app: gamemarket-backend
    spec:
      containers:
      - name: backend
        image: gamemarket/backend:latest
        ports:
        - containerPort: 8000
        env:
        - name: NODE_ENV
          value: "production"
        - name: DATABASE_URL
          valueFrom:
            secretKeyRef:
              name: database-secret
              key: url
        - name: REDIS_URL
          valueFrom:
            secretKeyRef:
              name: redis-secret
              key: url
        - name: RABBITMQ_URL
          valueFrom:
            secretKeyRef:
              name: rabbitmq-secret
              key: url
        - name: STEAM_API_KEY
          valueFrom:
            secretKeyRef:
              name: steam-secret
              key: api-key
        - name: JWT_SECRET
          valueFrom:
            secretKeyRef:
              name: jwt-secret

  name: gamemarket-backend
  namespace: gamemarket
```

## 监控与告警

### 系统监控

1. **Prometheus 配置**

```yaml
# prometheus.yml
global:
  scrape_interval: 15s

scrape_configs:
  - job_name: 'node'
    static_configs:
      - targets: ['node-exporter:9100']

  - job_name: 'backend'
    static_configs:
      - targets: ['gamemarket-backend:8000']

  - job_name: 'redis'
    static_configs:
      - targets: ['redis-exporter:9121']

  - job_name: 'postgres'
    static_configs:
      - targets: ['postgres-exporter:9187']
```

2. **Grafana 仪表盘**

- **系统概览**：CPU 使用率、内存使用率、磁盘使用率、网络流量
- **服务状态**：API 响应时间、请求成功率、并发连接数
- **数据库状态**：查询响应时间、连接数、缓存命中率
- **业务指标**：交易量、成交金额、用户活跃度

3. **告警规则**

```yaml
# alerting rules
groups:
- name: system
  rules:
  - alert: HighCPUUsage
    expr: (100 - (avg by(instance) (irate(node_cpu_seconds_total{mode="idle"}[5m])) * 100) > 80
    for: 5m
    labels:
      severity: warning
    annotations:
      summary: "High CPU Usage"
      description: "CPU usage is above 80% for 5 minutes"

  - alert: HighMemoryUsage
    expr: (node_memory_MemTotal_bytes - node_memory_MemAvailable_bytes) / node_memory_MemTotal_bytes * 100 > 80
    for: 5m
    labels:
      severity: warning
    annotations:
      summary: "High Memory Usage"
      description: "Memory usage is above 80% for 5 minutes"

- name: application
  rules:
  - alert: HighErrorRate
    expr: rate(http_requests_total{status=~"5.."}[5m]) / rate(http_requests_total[5m]) > 0.05
    for: 5m
    labels:
      severity: warning
    annotations:
      summary: "High Error Rate"
      description: "Error rate is above 5% for 5 minutes"

  - alert: SlowResponseTime
    expr: histogram_quantile(0.95, rate(http_request_duration_seconds_bucket[5m])) > 1
    for: 5m
    labels:
      severity: warning
    annotations:
      summary: "Slow Response Time"
      description: "95th percentile response time is above 1 second for 5 minutes"

- name: business
  rules:
  - alert: LowTradingVolume
    expr: sum(rate(trades_total[1h])) < 100
    for: 1h
    labels:
      severity: warning
    annotations:
      summary: "Low Trading Volume"
      description: "Trading volume is below 100 for 1 hour"

  - alert: HighDeliveryFailureRate
    expr: rate(delivery_failures_total[1h]) / rate(deliveries_total[1h]) > 0.1
    for: 1h
    labels:
      severity: critical
    annotations:
      summary: "High Delivery Failure Rate"
      description: "Delivery failure rate is above 10% for 1 hour"
```

### 日志管理

1. **ELK Stack 配置**

- **Elasticsearch**：存储和索引日志
- **Logstash**：处理和转换日志
- **Kibana**：可视化和分析日志

2. **日志格式**

```json
{
  "timestamp": "2023-01-01T00:00:00.000Z",
  "level": "info",
  "service": "backend",
  "module": "trade",
  "message": "Order matched",
  "data": {
    "orderId": 123,
    "symbol": "AKRED-FT-T1",
    "price": 100,
    "quantity": 5
  }
}
```

3. **日志索引策略**

- 按日期索引：`logs-2023.01.01`
- 设置索引生命周期：7天热数据，30天温数据，90天冷数据
- 实现日志轮转和清理，避免存储空间不足

### 告警通知

1. **通知渠道**

- **邮件**：发送详细的告警信息
- **短信**：发送紧急告警
- **Slack**：发送实时告警到团队频道
- **PagerDuty**：处理严重告警，确保及时响应

2. **告警级别**

- **Info**：信息性通知，不需要立即处理
- **Warning**：警告，需要关注但不紧急
- **Critical**：严重告警，需要立即处理

3. **告警处理流程**

1. 系统检测到异常，触发告警
2. 告警通过多种渠道发送给相关人员
3. 值班人员收到告警，进行初步分析
4. 如有必要，召集相关团队进行处理
5. 处理完成后，更新告警状态并记录处理过程

## 灾备与恢复

### 数据备份

1. **数据库备份**

- **全量备份**：每天执行一次，备份所有数据
- **增量备份**：每小时执行一次，备份自上次备份以来的变更
- **备份存储**：将备份存储在多个地理位置的存储服务中

2. **配置备份**

- 定期备份系统配置文件
- 备份 Kubernetes 配置和 Secrets
- 备份监控和告警配置

### 灾备方案

1. **多区域部署**

- 在多个 AWS 区域或阿里云区域部署服务
- 实现跨区域数据同步
- 配置 DNS 故障转移，当主区域不可用时自动切换到备用区域

2. **容器编排**

- 使用 Kubernetes 实现服务编排
- 配置 Pod 反亲和性，确保 Pod 分布在不同的节点上
- 配置水平自动扩缩容，根据负载自动调整副本数

3. **灾难恢复计划**

- 制定详细的灾难恢复计划，包括恢复步骤和时间目标
- 定期演练灾难恢复流程，确保团队熟悉恢复过程
- 建立灾难恢复时间目标（RTO）和恢复点目标（RPO）

### 业务连续性

1. **服务降级策略**

- 当系统负载过高时，自动降级非核心功能
- 优先保证交易和资产安全等核心功能
- 实现 graceful degradation，确保系统在部分功能不可用时仍能提供核心服务

2. **应急响应机制**

- 建立应急响应团队，明确角色和职责
- 制定应急响应流程，包括事件发现、评估、处理和恢复
- 建立应急联系人和沟通渠道，确保信息及时传递

3. **业务连续性测试**

- 定期进行业务连续性测试，验证系统在各种故障场景下的表现
- 测试包括：网络故障、数据库故障、服务器故障等
- 根据测试结果优化业务连续性计划

## 运维工具与脚本

### 部署脚本

1. **CI/CD 配置**

```yaml
# .github/workflows/deploy.yml
name: Deploy

on:
  push:
    branches:
      - main

jobs:
  build-and-deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      
      - name: Set up Node.js
        uses: actions/setup-node@v3
        with:
          node-version: '18'
      
      - name: Install dependencies
        run: npm install
      
      - name: Build frontend
        run: |
          cd frontend
          npm install
          npm run build
      
      - name: Build backend
        run: |
          cd backend
          npm install
          npm run build
      
      - name: Deploy to production
        run: |
          # 部署到 Kubernetes
          kubectl apply -f k8s/
```

2. **部署工具**

- **Helm**：管理 Kubernetes 应用的包管理器
- **Terraform**：基础设施即代码工具，管理云资源
- **Ansible**：配置管理工具，自动化部署和配置

### 监控脚本

1. **健康检查脚本**

```bash
#!/bin/bash

# 检查 API 服务
API_STATUS=$(curl -s -o /dev/null -w "%{http_code}" https://api.gamemarket.com/health)
if [ "$API_STATUS" -ne 200 ]; then
  echo "API service is down"
  exit 1
fi

# 检查数据库连接
DB_STATUS=$(pg_isready -h prod-db -U user)
if [ $? -ne 0 ]; then
  echo "Database connection failed"
  exit 1
fi

echo "All services are healthy"
exit 0
```

2. **性能监控脚本**

```bash
#!/bin/bash

# 收集系统性能数据
CPU_USAGE=$(top -bn1 | grep "Cpu(s)" | sed "s/.*, *\([0-9.]*\)%* id.*/\1/" | awk '{print 100 - $1}')
MEM_USAGE=$(free -m | awk '/Mem/{print $3/$2 * 100.0}')
DISK_USAGE=$(df -h | grep '/dev/sda1' | awk '{print $5}' | sed 's/%//')

# 发送到监控系统
echo "cpu_usage $CPU_USAGE" | curl -X POST --data-binary @- http://prometheus:9091/metrics/job/system_metrics
echo "mem_usage $MEM_USAGE" | curl -X POST --data-binary @- http://prometheus:9091/metrics/job/system_metrics
echo "disk_usage $DISK_USAGE" | curl -X POST --data-binary @- http://prometheus:9091/metrics/job/system_metrics
```

### 维护脚本

1. **数据库维护脚本**


2. **日志清理脚本**

```bash
#!/bin/bash

# 清理旧日志
find /var/log -name "*.log" -mtime +7 -delete

# 压缩较旧的日志
```

## 安全运维

### 安全审计

1. **定期安全扫描**

- 使用 OWASP ZAP 进行 Web 应用安全扫描
- 使用 Nessus 进行网络安全扫描
- 定期检查系统漏洞，及时更新补丁

2. **安全配置审计**

- 定期检查服务器安全配置
- 检查数据库访问控制
- 检查 API 访问权限

### 密钥管理

1. **密钥存储**

- 使用 AWS Secrets Manager 或阿里云 KMS 存储敏感信息
- 避免在代码中硬编码密钥
- 定期轮换密钥

2. **访问控制**

- 实现最小权限原则，只授予必要的权限
- 定期审查用户权限，移除不必要的权限
- 使用多因素认证，提高账户安全性

### 事件响应

1. **安全事件处理**

- 建立安全事件响应团队
- 制定安全事件响应计划
- 定期演练安全事件响应流程

2. **事件记录与分析**

- 记录所有安全事件
- 分析安全事件原因和影响
- 采取措施防止类似事件再次发生

## 总结

部署与运维是保证 Game Market 平台稳定运行的关键环节。通过合理的部署方案、完善的监控与告警机制、有效的灾备与恢复策略，可以确保平台在各种情况下都能正常运行，为用户提供可靠的交易服务。同时，通过安全运维措施，可以保护平台和用户的资产安全，建立用户信任。
